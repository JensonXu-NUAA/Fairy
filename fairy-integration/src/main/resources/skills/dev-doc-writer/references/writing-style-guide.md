# 语言风格规范（通用）

本文件定义所有语言通用的文档写作风格。语言特定的代码块规范见各语言规范文件。

---

## 一、整体语气与立场

**技术叙述，不是用户手册**。读者是参与开发的工程师，语气客观、简练、精确，不需要引导性语言。

**有主语，不空洞**。每句话都要有明确主语（组件名、方法名、机制名），避免无主语的被动句。

**说决策，不只说现象**。不只描述"做了什么"，要说明"为什么这样做"。

示例对比：
- 弱：`UserService 缓存了用户数据`
- 强：`UserService 将用户数据缓存在内存 Map 中，避免每次请求都触发数据库查询`

---

## 二、词汇选择

**描述职责定位**：
- 类/模块：`是…的核心`、`负责…`、`定义了…的统一契约`、`是…的入口`
- 接口/抽象：`定义了…`、`约定了…`
- 工厂/构建器：`组装`、`构建`、`创建`
- 管理器/注册表：`维护`、`管理`、`路由`、`分发`
- 监听/响应：`监听`、`感知`、`响应`

**禁止使用的空洞表达**：

| 禁止 | 替换 |
|---|---|
| 该组件提供了…功能 | 该组件负责… / 该组件定义了… |
| 可以用来… | 用于… |
| 非常重要的是… | 直接陈述 |
| 值得注意的是… | 直接陈述 |
| 通过以上介绍… | 无需总结句 |
| 该组件的主要作用是 | 该组件是… / 该组件负责… |
| 简单来说… | 直接说 |

---

## 三、职责说明三句模板

每个组件的职责说明控制在 1–3 句话，按以下结构组织：

**第一句（必须）：定性句**
```
XxxComponent 是 [定位]，[核心职责]。
XxxComponent 负责 [动作]，[目的/结果]。
```

**第二句（按需）：设计决策句**
```
原因在于 [技术原因]，通过 [方案] 可以 [效果]。
使用 [技术选择] 而非 [替代方案]，原因在于 [理由]。
通过 [方式]，[解决什么问题]。
```

**第三句（按需）：协作关系句**
```
依赖 [XxxComponent] 完成 [具体操作]。
被 [XxxComponent] 调用，返回 [结果类型]。
新增 [扩展点] 时只需 [最小改动]，符合 [设计原则]。
```

## 四、背景需求写法

一段话，100–200 字，依次覆盖：

1. **驱动因素**：外部变化（业务增长、技术演进、需求复杂化）→ 引出为什么需要这个模块
2. **定位声明**：`为此，[项目/团队] 以 [核心组件] 为核心，[整合了什么能力]，[服务于什么目标]`
3. **约束与目标**：`该模块需支持 [特性A]、[特性B]，兼顾 [质量维度]，是 [系统定位]`

不分点，不用项目符号，保持段落流畅。

## 五、数据流描述风格

使用伪代码 + ASCII 树形符号，规则如下：

```
调用方
  │  动作描述（如 POST /api/xxx，或 调用 fn()）
  ▼
被调用组件
  ├─① 步骤描述
  ├─② 步骤描述
  │   ├── 子步骤 A：说明
  │   └── 子步骤 B：说明
  └─③ 步骤描述
      └── 返回结果
```

符号含义：
- `▼`：调用进入下一层
- `│`：当前层逻辑继续
- `├─①②...`：有序步骤
- `→`：同步调用并返回结果
- `⇢`（或注明"异步"）：异步调用

分支逻辑用缩进 + 条件说明：
```
├── 条件成立（如 hasToolCalls()）→ 推送 event_A
└── 条件不成立               → 跳过
```

## 六、格式细节

- **组件名、方法名、字段名**：正文中用反引号，不加粗
- **配置项、常量、文件路径**：用反引号
- **数字**：用阿拉伯数字，不用文字（"3 个"，不是"三个"）
- **括号用途**：仅用于补充缩写含义、举例、技术背景，不用于核心陈述
- **"等"字**：列举不完全时使用，列举完全时不用

## 七、篇幅控制

| 章节 | 字数/行数参考 |
|---|---|
| 背景需求 | 100–200 字，1 段 |
| 每个组件职责说明 | 50–150 字，1–3 句 |
| 数据流 | ≤ 40 行伪代码 |
| 整篇文档 | 1500–3000 字（复杂架构可到 4000） |

超过 5000 字应考虑拆分为多个文档。
